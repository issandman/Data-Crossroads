# AI Coding 正在重新发现控制论

> 2026-06-12

## 从 Prompt 到 Loop：行业在重新发现什么

过去两年 AI Coding 经历了三个阶段。

第一阶段是 Prompt Engineering，研究怎么跟模型说话。后来发现模型越强，Prompt 的边际收益越低。第二阶段是 Context Engineering，意识到真正决定效果的不是怎么说，而是给模型看什么。RAG、Memory、MCP、Tool Calling 都属于这一层。第三阶段是 Loop Engineering，发现即使上下文给对了，Agent 仍然不稳定，因为软件开发本质上不是一次推理，而是一个持续反馈过程：写代码、编译、测试、发现问题、修改、再测试。这是一个循环，不是一次调用。

有趣的是，如果回头去翻维纳 1948 年的《控制论》，会发现这条演化路径并不新鲜。控制论的核心就一句话：通过信息的获取与反馈，对系统进行调节和控制，以实现特定目标。设定目标、感知偏差、施加干预、消除偏差，这就是恒温器的工作原理，也是 Kubernetes Controller 的工作原理，也是今天 Agent Loop 的工作原理。

三者是同一棵家谱树上的不同分支。AI Coding 不是在凭空发明新范式，而是在重新发现一个七十多年前就被严格论证过的旧思想。

## K8s 其实只做了一件事

很多人以为 Kubernetes 是容器编排平台。从使用角度确实如此，但从设计哲学上看，K8s 真正做的事情只有一件：把控制论带进了软件基础设施。

传统运维是命令式的，你告诉系统每一步怎么做：启动 Pod1、启动 Pod2、启动 Pod3。K8s 是声明式的，你只说“我要 3 个副本”，写成一份 YAML 提交，然后不管了。系统内部有一群 Controller，永远在跑同一个循环：读取期望状态，观察实际状态，发现不一致就纠正。Pod 挂了自动补，节点宕了自动迁。

这种自愈能力不是什么高级功能，而是负反馈回路的自然结果。

K8s 还有一条极其重要的设计决策：Controller 不直接信任单次事件通知。它不是依赖“我有没有收到那个事件”，而是以 API Server 中可重新观测的资源状态为准，通过 informer、cache 和 work queue 不断触发 reconcile。进程崩溃、网络丢包、错过一条消息，都不会天然破坏系统；只要后续还能重新观测到当前状态，Controller 就有机会继续把实际状态拉回期望状态。用术语说，这是 level-triggered（基于状态触发）的思路，而不是单纯依赖 edge-triggered（基于事件边沿触发）。

## Agent Loop 为什么长得越来越像 Reconciliation Loop

把 K8s Controller 的执行过程和 Agent Loop 放在一起看：

```text
K8s：读取期望状态 -> 观测集群 -> 计算 diff -> 创建/删除/更新资源 -> 循环
Agent：理解目标 -> 读取代码库 -> 评估是否达标 -> 修改代码 -> 运行测试 -> 循环
```

结构几乎完全一致。而且映射关系越深挖越精确：

Goal 对应 Spec，Evaluation 对应 Status，Agent 对应 Controller，Git 仓库对应持久状态存储。Agent 社区说“Agent 会忘，仓库不会忘”，把状态从对话历史搬到 `AGENTS.md`、`memory.md`、`progress.md` 等文件里，这就是 K8s 十几年前就内化的经验：不信任易失的对话历史，只信任可重新观测的持久状态。Maker/Checker 分离对应 Admission Controller，两者存在的理由都是同一个：不要相信执行器的自我评价。

Prompt 正在变成 Spec。从“打开文件 A、修改函数 B、新增测试 C”变成“修复支付系统的并发问题，确保测试全部通过”。你不告诉 Agent 怎么做，只告诉它你要什么结果。这和 K8s 的写法本质上一样，你不会告诉 Controller “启动 Pod1、启动 Pod2、启动 Pod3”，你只写：

```yaml
replicas: 3
```

## 约束、迭代、传感器：控制论如何解释实践经验

有三条实践经验，过去我只知道“管用”，控制论给出了“为什么管用”的解释。

**第一条：约束的本质是压缩可能性空间。** 让 AI 写一个排序，它有一万种写法，这就是可能性空间 M。指定语言，M 缩小；指定框架，继续缩小；指定命名规范、架构分层、代码规范，M 被压缩到 m。每一条约束都在帮 Agent 收敛，而不是在“限制 AI 的能力”。这也给了我一个判断规则质量的标准：一条 `SKILL` 或 `RULE`，如果不能可衡量地缩小可能性空间，它就是噪声。

**第二条：多轮迭代不是权宜之计，是数学必然。** 控制论里，任何控制器的单次控制力都有上限。就像老鹰抓兔子，不可能一次俯冲就抓到，必须通过多次俯冲累积控制力。Agent 再强也一样，单次推理只能把 M 压缩到 M1，后续还需要多轮迭代从 M1 到 m。所以 Loop Engineering 不是“模型不够好的临时方案”，而是有限控制力下的必然架构选择。

**第三条：没有传感器就没有闭环。** 控制的起点是偏差：

```text
Error = Goal - Observed State
```

如果测不出 `Observed State`，偏差就不存在，控制也无从发生。传感器分两类：业务无关的，例如编译检查、lint、安全扫描，不需要懂业务就知道代码有没有语法错误；业务相关的，例如基于业务规则的测试，需要人把规则写到可机器验证的程度。一个没有传感器的 Agent Loop 就是一个没有温度计的恒温器，只会空转。

还有一个值得警惕的现象：坏代码在 AI 时代是自繁殖系统。控制论里，自繁殖系统一旦启动会指数增长，核爆炸、癌细胞、传染病都是这个模式。AI 参考已有代码生成新代码，一次烂实现会被无数次复制放大。这比“技术债”严重得多：技术债是线性累积，自繁殖系统是指数爆炸。过去跳过规范的代价可能在你离职后才爆发，AI 时代可能几天就失控。所以必须在正反馈失控之前引入负反馈：代码规范检查、自动重构、定期扫描偏差。

## 类比的边界：一道无法跨越的鸿沟

以上都是两者的相似之处。但如果只讲相似，就成了事后附会。真正有价值的问题是：两者的差异在哪，以及这些差异意味着什么。

控制论有一个前提概念叫被控对象模型（Plant Model），你得大致知道“推一下，系统会怎么动”。K8s 相对更接近这个前提：创建一个 Pod、删除一个 Pod、修改一个 Deployment，这些动作的结果大体确定、可预测，也更容易写成幂等逻辑。而 Agent 的执行器是概率云。同样的指令，模型今天可能这样写，明天可能那样写；可能更好，也可能完全跑偏。

这一差异导致了一连串后果。

写得好的 K8s controller，在目标清晰、操作幂等、外部条件稳定时，通常具备可预期的收敛性；Agent Loop 则没有同等级别的收敛保证。它可能震荡，改过去又改回来；可能卡死，反复报同一个错但换不了思路；也可能谎称完成，给自己打满分但代码根本跑不通。K8s 判定完成常常可以落到 `actual == desired`，Agent 的“完成”却常常不可机器判定，要靠测试、类型检查，甚至另一个 LLM 做裁判，而裁判本身还会错。K8s Operator 的动作空间相对受限，通常围绕已知资源做增删改；Agent 的动作空间近乎无界，它可以写任意代码、执行任意 shell、修改任意文件。

所以 Agent 领域需要 Maker/Checker 分离、需要独立模型做裁判、需要强制测试套件做硬门槛、需要人类在环。这些不是 K8s 常规 controller 的核心问题。K8s 用了很多年，靠确定性接口、幂等设计和大量测试，才挣到“无人值守”的信任。Agent Loop 现阶段更像一个“还没挣到自主权”的 Operator：同样是调和循环架构，但因为执行器不可靠，上面必须再套一层人类监督回路。

## 真正的天花板不是 Loop，而是 Goal 和 Evaluation

想清楚上面的差异之后，会发现一件更根本的事情：大家把注意力放在 Loop 的设计上，但真正决定上限的，是 Goal 能不能被精确定义、Evaluation 能不能被机器执行。

K8s 之所以能做到高度自动化，根本原因不是 Controller 写得有多神，而是 `replicas: 3` 这种目标没有歧义，`actual_replicas == 3` 这种判定没有模糊空间。Goal 和 Evaluation 被形式化了，收敛才成为可能。

AI Coding 的问题恰好在这里。“优化支付系统”是什么意思？性能？稳定性？安全性？可维护性？不同人理解完全不同。Goal 本身就是模糊的，Evaluation 也很难做到无歧义。

但我认为这里存在一个硬边界，不是所有 Goal 都能被形式化。基础设施目标，例如副本数、延迟、可用率，天然更容易形式化；软件工程中最难的决策，例如架构取舍、可维护性、设计方向，本质上很难完全形式化。前者可以写成类似 YAML spec 的东西让 Agent 自主收敛，后者仍然需要人类判断。这不只是暂时的技术限制，也和问题本身的性质有关。Agent 能自主到什么程度，取决于这条边界在哪里。

类似地，“模型只是可替换组件”这个判断方向是对的，但现在下结论还早。K8s 的 Pod 更接近同质资源，一个 Nginx Pod 和另一个 Nginx Pod 没有本质区别，所以“Controller 比 Pod 重要”成立。但现阶段不同模型在编码循环中的表现差异仍然巨大，远不是完全可替换。

还有一个常被忽略的约束：成本。K8s reconcile 的计算成本很低，可以频繁触发。Agent loop 每轮都是一次昂贵的 LLM 调用。“多跑几轮自然收敛”这种 K8s 式的乐观策略，在 Agent 领域的经济可行性完全不同。

## 对我的实际影响

这些思考落到实践上，影响最大的是一个观念转变：**写规则不是浪费时间，那就是这个时代的本职工作。**

回头看我们做 Metric Hermes 的过程，花在 `guardrails rules`、`static knowledge base`、`table routing registry` 这些东西上的时间，远超写 Agent 核心逻辑的时间。那些东西就是约束，用来压缩可能性空间；也是传感器，用来让闭环成为可能。它们不是 Agent 的外围配置，而是系统的控制面。代码生成只是执行器，真正决定系统稳定性的，是这些看起来“不写业务代码”的东西。

工程师的核心价值正在从“实现”转向“评估”。过去你思考的是函数怎么写，现在你思考的是：目标怎么定义才能让 Agent 收敛，验证怎么设计才能让“完成”可以机器判定，状态怎么管理才能让循环可以随时重启而不丢进度。

不是程序员不需要了。是工作内容变了，从划桨的人变成了设计调速器的人。
